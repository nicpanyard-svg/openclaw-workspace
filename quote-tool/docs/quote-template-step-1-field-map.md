## 1. High-Level Template Shape

The sample is a **10-page branded proposal** with this overall pattern:

1. **Cover page**
2. **Executive Summary page**
3. **Pricing page — Section A / Pool pricing**
4. **Pricing page — Section B / Starlink equipment pricing**
5. **Pricing page — Section C / Wi-Fi equipment pricing**
6. **Grand total + addendum intro**
7. **Acceptance criteria table**
8. **Remediation / acceptance legal text**
9. **General service terms + pricing terms**
10. **Documentation details & tracking**

For the quote tool v1 requested by Nick:
- **Keep the same general branded look and document feel**
- **Core functional quoting scope is Section A + Section B**
- Other pages/sections can remain locked/static unless explicitly made configurable later

## 5. Repeating Structures

These are the repeating structures the app/template engine must support.

## 5.1 Section A pricing rows
Depending on mode:
- Pool rows repeat
- Per-kit rows repeat or render as a fixed small set

Recommended model:
- `sectionA.rows[]`
  - enough flexibility to render either pricing mode in the same table component

## 5.2 Section B equipment rows
This is the main repeating commercial table.

Recommended model:
- `sectionB.lineItems[]`
- unlimited row support
- total row appended automatically

## 5.3 Documentation / revision rows
Sample has revision details table with version/change rows.

Recommended model:
- `revisionHistory[]`
  - `version`
  - `changeDetails`

V1 can default to one row.

## 9. Decisions Confirmed by This Mapping

- The quote tool should be a **separate internal app**, not part of Mission Control
- The sample proposal provides a strong template basis with a clear page system
- **Section A** is the service-pricing area and should render **either Pool or Per Kit pricing**
- **Section B** is the one-time Starlink hardware table
- **Section C exists in the sample** but should not block v1
- The template needs support for both **standard catalog** and **custom** hardware rows
- The sample’s branded look can be preserved without preserving every non-core commercial section as dynamic in v1
