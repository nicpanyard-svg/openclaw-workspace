# Recovery Plan — Mission Control / iNet World / Graham Board

## The real issue
Status keeps sounding the same because too much work has been hidden in setup, structure, and partial systems.

Nick cannot manage by vibes. He needs visible proof:
- what was built
- what now works
- what is blocked
- who owns the next move

So the fix is simple: **build the minimum tracking layer, then push visible product output every day.** Mission Control supports reality; it does not replace reality.

---

## Operating rule for all 3 lanes
1. **No stop-and-wait.** If blocked, do the next best useful step.
2. **Major blockers surface immediately.** Do not let blockers sit quietly.
3. **Show output, not effort.** A page, a board, a working flow, a populated view, a screenshot, a checklist moved to done.
4. **Cut side polish.** No extra architecture, styling, or framework work unless it directly unlocks visible progress.
5. **Daily proof standard:** every lane must produce something Nick can see in under 60 seconds.

---

# 1) Mission Control

## Plain issue
Mission Control is not useful enough yet to track reality, but it has consumed enough time that Nick expects it to already be doing that job.

Right now it is in the danger zone: too incomplete to manage from, but heavy enough to slow everything else down.

## Build order
1. **Build only the core tracking view**
   - Lane name
   - Owner
   - Current task
   - Status: Not started / In progress / Blocked / Done
   - Last visible deliverable
   - Current blocker
   - Next step
2. **Add one clean daily update flow**
   - Each lane updates once per check-in window
   - No essays
   - No duplicate notes in multiple places
3. **Make blocker visibility obvious**
   - Blocked items grouped at top or clearly marked
4. **Only after that, add convenience**
   - build calendar
   - nicer layout
   - extra views
   - historical reporting

## Ownership
- **Susan:** define structure, enforce update format, keep scope tight
- **Jill:** update execution status for product work in plain language
- **Jack:** stabilize access/supporting mechanics if Mission Control delivery depends on browser/system behavior
- **Graham:** update board-related lane status and visible outputs
- **Ike:** audit the updates, call out gaps, keep blocker log, translate messy status into plain truth for Nick

## Cut / defer for now
- Fancy UI work
- Extra dashboards
- Deep automation
- Reporting layers
- Any feature that does not help Nick answer: "What moved, what is blocked, what happens next?"

## Visible progress by next check-in
Nick should be able to open Mission Control and see:
- all 3 lanes present
- one owner per lane
- one active task per lane
- blocker state shown clearly
- one concrete deliverable listed per lane
- next step listed per lane

If that is not visible, Mission Control is still not ready.

# 2) iNet World

Nick does not need another explanation of the vision. He needs the worlds to work.

   - every tile present

- **Jack:** support/stabilize the browser, access, and technical path when tooling friction blocks progress

## Cut / defer for now
- Extra vertical expansion beyond the agreed active set
- Fancy transitions
- Deeper back-end logic not required for visible demo behavior
- Rework of already-good-enough scenes just for aesthetics

## Visible progress by next check-in
Nick should be able to see:
- the overview world/grid loads cleanly
- every active vertical tile is present
- every tile opens a non-blank detailed view
- the flood tile visual mismatch is resolved or clearly queued with a substitute in place

The target is not "better direction." The target is **working vertical count**.

Too much of Graham's work has been invisible setup. That may be necessary, but Nick cannot judge progress from hidden scaffolding.

This board needs to become a visible daily operating surface, not a research basement.

## Build order

   - richer indicators
   - broader automation

## Ownership

- Overbuilt scoring systems

# Check-in format Nick should get
For each lane, only this:
- **Owner**
- **What changed since last check-in**
- **What is working now**
- **What is blocked now**
- **What happens next**

No padded status. No repeated narrative.

---

# Blocker-handling rules
## Immediate rule
If a blocker will stop meaningful progress, surface it immediately to Ike/Nick.

## Then do this
1. State the blocker plainly
2. State impact plainly
3. State the workaround or next-best-step
4. Continue moving on that next-best-step until the blocker is cleared

## Bad behavior
- waiting quietly
- doing low-value busywork to look active
- repeating the same blocker without a workaround attempt

## Good behavior
- "Browser control is flaky. I completed the content prep and queued the exact next browser actions."
- "Mission Control is missing blocker visibility, so I updated the lane data manually first and will wire the display second."
- "One vertical is not finished, so I completed the other working tiles and flagged the remaining gap explicitly."

---

# Bottom line
Nick needs a system he can trust.

That means:
- **Mission Control gets cut down to a usable tracking core**
- **iNet World is judged by completed working verticals**
- **Graham board is judged by visible populated output, not setup**

The team should be managed on **visible movement per lane**, not effort stories.
